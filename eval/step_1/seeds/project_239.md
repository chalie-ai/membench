## 1. EXECUTIVE SUMMARY

### 1.1 Project Overview
Project "Pinnacle" represents a critical strategic pivot for Deepwell Data. The objective is the complete rebuild of the company’s healthcare records platform, specifically tailored for the aerospace industry. While aerospace is primarily associated with aviation and space exploration, the operational requirements for astronaut health monitoring, high-G physiological tracking, and long-term orbital health records necessitate a highly specialized healthcare record system.

The current legacy system has suffered "catastrophic" user feedback. Clients have reported critical failures in data latency, a non-intuitive UI that hampers emergency medical response, and a fragile data ingestion pipeline. Pinnacle is not merely an update; it is a full-scale replacement designed to restore client trust and secure Deepwell Data’s position as the primary healthcare data provider for aerospace contractors and government agencies.

### 1.2 Business Justification
The aerospace sector demands zero-fail reliability. The legacy system's failure is not just a UX issue—it is a compliance and safety risk. In the aerospace context, the inability to rapidly retrieve a pilot’s medical history during a flight anomaly can have fatal consequences. Furthermore, the existing system lacks the scalability to handle the massive telemetry datasets generated by modern biometric sensors used in high-altitude flights.

By migrating to a modern stack (Elixir/Phoenix), Pinnacle will provide the concurrency and fault tolerance necessary to handle thousands of simultaneous real-time data streams. The move to a modular monolith allows the team to maintain velocity while preparing for an eventual transition to microservices as the client base scales.

### 1.3 ROI Projection
The financial justification for Pinnacle is rooted in two primary vectors: churn reduction and operational efficiency. 

1. **Churn Reduction:** Currently, Deepwell Data faces a projected 25% churn rate of aerospace clients due to the legacy system's instability. Successfully delivering Pinnacle is estimated to retain approximately $1.2M in Annual Recurring Revenue (ARR).
2. **Operational Cost Reduction:** The legacy system is built on a resource-heavy, monolithic Java framework with inefficient database queries. The transition to Elixir/Phoenix and a refined PostgreSQL schema is projected to reduce the cost per transaction by 35%. This is achieved through better utilization of CPU cores and reduced memory overhead per connection.
3. **Market Expansion:** FedRAMP authorization will open the door to direct US Government contracts, potentially increasing the Total Addressable Market (TAM) by 40% within 24 months.

With a modest budget of $400,000, the ROI is projected to be positive within 14 months of the Milestone 3 stability confirmation.

## 2. TECHNICAL ARCHITECTURE

### 2.1 Core Stack
*   **Language/Framework:** Elixir / Phoenix (v1.7+)
*   **Frontend:** Phoenix LiveView (for real-time updates and state management without heavy JS overhead)
*   **Database:** PostgreSQL 15 (Relational data, JSONB for flexible medical record schemas)
*   **Deployment:** Fly.io (Global distribution, low latency)
*   **CI/CD:** GitHub Actions (Automated testing and deployment)
*   **Infrastructure:** Modular Monolith $\rightarrow$ Microservices (Incremental)

### 2.2 Architectural Philosophy
Pinnacle utilizes a "Modular Monolith" approach. The codebase is organized into strictly bounded contexts (e.g., `Pinnacle.Auth`, `Pinnacle.Records`, `Pinnacle.Notifications`). This ensures that while the application is deployed as a single unit for simplicity and cost-effectiveness, the internal boundaries are clean. When a specific module (such as the Notification System) exceeds the resource capacity of the monolith, it can be extracted into a separate Elixir node or a microservice with minimal refactoring.

                                                                 |

                                         |                     |                |
                                         v                     v                v
                                [ REDIS CACHE ] <---> [ POSTGRESQL DB ] <---> [ S3 BUCKET ]

### 2.4 Deployment Strategy
We employ a **Blue-Green Deployment** strategy via GitHub Actions and Fly.io. 
1. **Green (Staging):** New code is deployed to a staging environment where Lina Jensen (QA Lead) performs regression testing.
2. **Blue (Production):** Once validated, traffic is shifted incrementally from the old version to the new version.
3. **Rollback:** If the error rate exceeds 0.5% in the first ten minutes, the load balancer automatically reverts to the previous stable build.

### 2.5 Security & Compliance
Given the industry, **FedRAMP authorization** is non-negotiable. 
*   **Encryption:** AES-256 for data at rest; TLS 1.3 for data in transit.
*   **Audit Logging:** Every read/write access to a patient record is logged to an immutable table (`audit_logs`).
*   **Identity Management:** Integration with government-approved MFA providers.

## 3. DETAILED FEATURE SPECIFICATIONS

### 3.1 Notification System (Priority: High | Status: In Design)
**Description:** A multi-channel communication engine designed to alert aerospace medical officers of critical health anomalies in real-time.

**Functional Requirements:**
*   **Multi-Channel Delivery:** The system must support Email (via SendGrid), SMS (via Twilio), In-App (via LiveView), and Push Notifications (via Firebase).
*   **Priority Queueing:** Notifications are categorized as *Critical* (immediate delivery), *Warning* (15-minute delay allowed), and *Informational* (batched).
*   **User Preferences:** A granular preference center where users can toggle which channels are used for specific alert types.
*   **Escalation Logic:** If a *Critical* notification is not acknowledged in-app within 5 minutes, the system must automatically escalate to SMS and then to a phone call.

**Technical Implementation:**
The system will utilize `Oblique` or a custom GenServer-based queue to handle asynchronous delivery. The `notifications` table will track the status of every single alert. To avoid blocking the main application thread, notifications will be dispatched to a background job processor (e.g., `Oban`).

**Acceptance Criteria:**
*   Critical alerts must reach the end-user within $\le 30$ seconds.
*   The system must handle 10,000 concurrent notifications without degrading the primary record-viewing performance.
*   Audit logs must prove the exact timestamp of delivery for compliance.

### 3.2 Real-time Collaborative Editing (Priority: High | Status: Not Started)
**Description:** A system allowing multiple healthcare providers to update a patient's record simultaneously during an active medical event (e.g., a mid-flight emergency).

**Functional Requirements:**
*   **Concurrent Editing:** Multiple users should be able to edit different fields of the same record without overwriting each other.
*   **Presence Indicators:** Users must see who else is currently viewing or editing the record (cursor tracking).
*   **Conflict Resolution:** Use of Operational Transformation (OT) or Conflict-free Replicated Data Types (CRDTs) to merge changes.
*   **Version History:** A complete "diff" history of every change made during a collaborative session.

**Technical Implementation:**
This will be implemented using **Phoenix Presence** for user tracking and a CRDT-based approach (likely utilizing `LWW-Element-Set` logic) to handle field-level updates. The state will be synchronized over WebSockets. When a conflict occurs at the exact same millisecond on the same field, the "Last Write Wins" (LWW) strategy will be applied based on a high-precision server timestamp.

**Acceptance Criteria:**
*   Latency between user updates must be $< 200\text{ms}$.
*   No data loss during simultaneous edits of different fields.
*   Correct attribution of which user made which change in the history log.

### 3.3 Webhook Integration Framework (Priority: Low | Status: In Design)
**Description:** An extensible framework allowing third-party aerospace telemetry tools (e.g., heart rate monitors, O2 sensors) to push data directly into Pinnacle.

**Functional Requirements:**
*   **Custom Endpoints:** Ability for admins to create unique webhook URLs for different hardware providers.
*   **Payload Validation:** Support for JSON and XML payloads with schema validation.
*   **Retry Logic:** Exponential backoff for failed deliveries from the provider.
*   **Security:** HMAC signature verification to ensure the data comes from a trusted source.

**Technical Implementation:**
A dynamic routing module will be created. When a request hits `/api/webhooks/:provider_id`, the system will look up the associated `webhook_config`, verify the signature using a stored secret, and then pass the payload to a specific "Parser" module designed for that provider.

**Acceptance Criteria:**
*   Ability to onboard a new third-party provider via configuration without deploying new code.
*   Maximum processing time for webhook ingestion $< 500\text{ms}$.

### 3.4 A/B Testing Framework (Priority: Low | Status: Complete)
**Description:** A system baked into the feature flag infrastructure to test new UI components or workflows on a subset of users.

**Functional Requirements:**
*   **Cohort Assignment:** Ability to assign users to "Control" or "Test" groups based on percentage or user ID.
*   **Flag Management:** A dashboard to toggle flags on/off in real-time without a restart.
*   **Metric Tracking:** Integration with the analytics engine to compare the performance of Version A vs. Version B.

**Technical Implementation:**
This was implemented using a combination of a `feature_flags` table and a cached `ETS` (Erlang Term Storage) table for rapid lookups. The logic resides in the `Pinnacle.Experiment` module, which intercepts request pipelines to inject specific CSS classes or LiveView templates based on the user's assigned bucket.

**Acceptance Criteria:**
*   Zero impact on page load time ($< 10\text{ms}$ overhead).
*   Correct persistent assignment (User X always sees Version B).

### 3.5 Data Import/Export (Priority: Low | Status: In Review)
**Description:** A tool to migrate records from legacy systems and export them for government audits, featuring automatic format detection.

**Functional Requirements:**
*   **Auto-Detection:** The system must analyze the first 100 lines of an uploaded file to determine if it is CSV, JSON, HL7, or FHIR.
*   **Mapping Tool:** A UI for users to map columns from the uploaded file to Pinnacle's internal schema.
*   **Bulk Export:** Ability to export millions of records into a compressed ZIP file containing CSVs.

**Technical Implementation:**
The import logic uses a "Streaming Upload" approach to avoid memory spikes. It utilizes `NimbleCSV` for parsing and a custom regex-based detection engine for format identification. Exporting is handled by an asynchronous `Oban` job that streams the database cursor directly to an S3 bucket.

**Acceptance Criteria:**
*   Successful import of a 1GB file without crashing the application node.
*   Accuracy of format detection $\ge 98\%$.

## 4. API ENDPOINT DOCUMENTATION

All endpoints are versioned under `/api/v1/`. All requests require a Bearer Token in the header.

### 4.1 `GET /api/v1/records`
**Description:** Retrieve a paginated list of healthcare records.
*   **Query Params:** `page` (int), `limit` (int), `patient_id` (uuid).
*   **Response (200 OK):**
    ```json
    {
      "data": [
        { "id": "rec_123", "patient_id": "pat_456", "created_at": "2023-10-01T10:00Z" }
      ],
      "meta": { "total": 150, "page": 1 }
    }
    ```

### 4.2 `POST /api/v1/records`
**Description:** Create a new medical record.
*   **Request Body:**
    ```json
    {
      "patient_id": "pat_456",
      "clinical_notes": "Patient exhibited tachycardia during G-force test.",
      "vitals": { "heart_rate": 110, "bp": "140/90" }
    }
    ```
*   **Response (201 Created):** `{ "id": "rec_789", "status": "saved" }`

### 4.3 `PATCH /api/v1/records/:id`
**Description:** Update a specific record.
*   **Request Body:** `{ "clinical_notes": "Updated: Patient stabilized." }`
*   **Response (200 OK):** `{ "id": "rec_789", "updated_at": "2023-10-24T12:00Z" }`

### 4.4 `GET /api/v1/patients/:id/history`
**Description:** Fetch full medical history for a specific astronaut.
*   **Response (200 OK):** `[{ "date": "2022-01-01", "event": "Centrifuge Test", "result": "Pass" }]`

### 4.5 `POST /api/v1/notifications/send`
**Description:** Manually trigger a notification.
*   **Request Body:** `{ "user_id": "usr_111", "channel": "sms", "message": "Urgent: Review Record 789" }`
*   **Response (202 Accepted):** `{ "job_id": "job_abc123" }`

### 4.6 `GET /api/v1/webhooks/verify`
**Description:** Test the connection of a third-party webhook.
*   **Response (200 OK):** `{ "status": "connected", "latency": "45ms" }`

### 4.7 `POST /api/v1/import/detect`
**Description:** Upload a snippet of a file to detect its format.
*   **Request Body:** Multipart form-data (file snippet).
*   **Response (200 OK):** `{ "detected_format": "HL7", "confidence": 0.99 }`

### 4.8 `DELETE /api/v1/records/:id`
**Description:** Soft-delete a record for compliance.
*   **Response (204 No Content):** Empty body.

## 5. DATABASE SCHEMA

The database uses PostgreSQL 15. All tables use `UUID` as primary keys.

### 5.1 Table Definitions

1.  **`users`**: Core user accounts.
    *   `id` (UUID, PK), `email` (String), `password_hash` (String), `role` (Enum: admin, medic, observer), `created_at` (Timestamp).
2.  **`patients`**: Astronaut/Pilot identity.
    *   `id` (UUID, PK), `full_name` (String), `date_of_birth` (Date), `blood_type` (String), `clearance_level` (Int).
3.  **`records`**: The primary medical data.
    *   `id` (UUID, PK), `patient_id` (FK $\rightarrow$ patients), `author_id` (FK $\rightarrow$ users), `content` (JSONB), `version` (Int), `created_at` (Timestamp).
4.  **`record_versions`**: History for the collaborative editing system.
    *   `id` (UUID, PK), `record_id` (FK $\rightarrow$ records), `diff` (JSONB), `user_id` (FK $\rightarrow$ users), `timestamp` (Timestamp).
5.  **`notifications`**: Tracking for all outgoing alerts.
    *   `id` (UUID, PK), `user_id` (FK $\rightarrow$ users), `channel` (String), `status` (Enum: pending, sent, failed), `priority` (Int).
6.  **`notification_logs`**: Immutable delivery proof.
    *   `id` (UUID, PK), `notification_id` (FK $\rightarrow$ notifications), `delivery_timestamp` (Timestamp), `provider_response` (Text).
7.  **`webhook_configs`**: Configuration for 3rd party integrations.
    *   `id` (UUID, PK), `provider_name` (String), `secret_key` (String), `target_endpoint` (String), `is_active` (Boolean).
8.  **`feature_flags`**: Store for A/B testing and toggles.
    *   `id` (UUID, PK), `flag_key` (String, Unique), `is_enabled` (Boolean), `percentage_rollout` (Int).
9.  **`user_flag_assignments`**: Which user gets which A/B version.
    *   `user_id` (FK $\rightarrow$ users), `flag_id` (FK $\rightarrow$ feature_flags), `assigned_group` (String: A/B).
10. **`audit_logs`**: FedRAMP compliant access logs.
    *   `id` (UUID, PK), `user_id` (FK $\rightarrow$ users), `action` (String), `resource_id` (UUID), `ip_address` (Inet), `timestamp` (Timestamp).

### 5.2 Relationships
*   `patients` $\rightarrow$ `records` (One-to-Many)
*   `records` $\rightarrow$ `record_versions` (One-to-Many)
*   `users` $\rightarrow$ `notifications` (One-to-Many)
*   `users` $\rightarrow$ `audit_logs` (One-to-Many)

## 6. DEPLOYMENT AND INFRASTRUCTURE

### 6.1 Environment Strategy
Pinnacle utilizes three distinct environments to ensure stability before reaching government clients.

#### 6.1.1 Development (Dev)
*   **Purpose:** Active feature development and unit testing.
*   **Infrastructure:** Local Docker containers for PostgreSQL and Redis; Fly.io "Dev" app.
*   **Data:** Anonymized synthetic data.
*   **Deployment:** Triggered on push to `develop` branch.

#### 6.1.2 Staging (Stage)
*   **Purpose:** QA validation, UAT (User Acceptance Testing), and FedRAMP pre-audit.
*   **Infrastructure:** Mirror of Production architecture on Fly.io.
*   **Data:** Scrubbed production snapshots (GDPR/HIPAA compliant).
*   **Deployment:** Triggered on merge to `release/v*` branches.

#### 6.1.3 Production (Prod)
*   **Purpose:** Live aerospace medical operations.
*   **Infrastructure:** Multi-region Fly.io deployment with high-availability PostgreSQL clusters.
*   **Data:** Encrypted live patient data.
*   **Deployment:** Manual trigger from Staging $\rightarrow$ Production via Blue-Green shift.

### 6.2 Infrastructure as Code (IaC)
All infrastructure is managed via Terraform. This ensures that the environment can be recreated in a different region if a provider outage occurs, maintaining the 99.9% uptime requirement.

## 7. TESTING STRATEGY

### 7.1 Unit Testing
*   **Focus:** Pure functions, business logic in Elixir modules, and schema constraints.
*   **Tool:** `ExUnit`.
*   **Requirement:** Minimum 80% code coverage.
*   **Execution:** Run on every commit via GitHub Actions.

### 7.2 Integration Testing
*   **Focus:** Interaction between the Phoenix app and PostgreSQL, Redis, and external API mocks.
*   **Approach:** Use of `Mox` for mocking external services (Twilio, SendGrid).
*   **Requirement:** All critical paths (e.g., creating a record $\rightarrow$ triggering a notification) must have an integration test.

### 7.3 End-to-End (E2E) Testing
*   **Focus:** The user's perspective of the LiveView interface.
*   **Tool:** `Wallaby` or `Playwright`.
*   **Approach:** Simulate a user logging in, editing a record, and receiving an in-app notification.
*   **Requirement:** Critical "Happy Path" tests must pass before any merge to `main`.

### 7.4 Performance Testing
*   **Focus:** Ensuring the 35% cost-per-transaction reduction and system stability under load.
*   **Tool:** `K6` or `JMeter`.
*   **Scenario:** Simulate 500 concurrent users editing records and 1,000 concurrent webhook ingestions.

## 12. APPENDICES

### Appendix A: The "God Class" Debt Analysis
The legacy `SystemManager.ex` file contains 3,142 lines of code. It currently handles:
1.  **Authentication:** Session management, password hashing, and JWT issuance.
2.  **Logging:** All system errors are piped through a single `log_event` function.
3.  **Email:** Direct SMTP calls for every notification.

This creates a "Single Point of Failure." A bug in the email formatting logic can crash the entire authentication flow. The Pinnacle project replaces this with three distinct modules: `Pinnacle.Auth`, `Pinnacle.Logger`, and `Pinnacle.Notifier`.

### Appendix B: FedRAMP Authorization Checklist
To achieve authorization, the team must provide evidence for the following controls:
*   **AC-2 (Account Management):** Automated user onboarding/offboarding.
*   **AU-2 (Event Logging):** Implementation of the `audit_logs` table with non-repudiation.
*   **SC-7 (Boundary Protection):** Configuring Fly.io VPCs and security groups.
*   **CP-2 (Contingency Plan):** Proof of Blue-Green deployment and data backup recovery time objectives (RTO $< 4$ hours).
