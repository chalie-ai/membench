## 1. EXECUTIVE SUMMARY

### Project Overview
Project **Wayfinder** is a strategic IoT device network initiative commissioned by the Oakmount Group, a leading provider of legal services. The primary objective of Wayfinder is to consolidate four redundant internal tools—currently disparate systems handling device tracking, legal document auditing, hardware health monitoring, and client billing synchronization—into a single, unified platform. By collapsing these silos, Oakmount Group aims to eliminate operational redundancies, reduce the overhead of maintaining four separate codebases, and provide a "single pane of glass" for the firm's IoT infrastructure.

### Business Justification
The current operational model is unsustainable. With four separate tools, the legal services team is forced to perform manual data reconciliation across different databases, leading to a high rate of clerical error and significant latency in reporting. Each of the existing tools requires its own maintenance window, security patching, and licensing costs. The consolidation into Wayfinder represents a shift from a fragmented ecosystem to a centralized, event-driven architecture.

From a legal services perspective, the IoT network provides critical telemetry for high-value assets and secure document-handling hardware. Ensuring these devices are operational and synchronized with client billing records is paramount for revenue leakage prevention.

### ROI Projection
While the project is currently **unfunded**—bootstrapping with existing team capacity—the projected Return on Investment (ROI) is calculated through "cost avoidance" and "operational efficiency" metrics. 

1. **Direct Cost Reduction:** Elimination of three redundant SaaS licenses and cloud hosting overhead for the deprecated tools, estimated at **$145,000 per annum**.
2. **Labor Efficiency:** Reduction in manual data entry and reconciliation time for the legal administrative staff. We project a saving of 40 man-hours per week across the department, valued at approximately **$110,000 per annum**.
3. **Risk Mitigation:** By moving to an ISO 27001 certified environment, the firm reduces the risk of compliance fines related to legal data mishandling, which can reach millions of dollars in the legal sector.

The total projected first-year savings upon full production launch (2026-06-15) is **$255,000**, with an expected increase in efficiency resulting in an NPS lift from the current baseline.

## 2. TECHNICAL ARCHITECTURE

### High-Level Architecture
Wayfinder is built on a microservices architecture to ensure scalability and fault tolerance. The core logic is implemented in **Python/Django**, utilizing **PostgreSQL** for relational data and **Redis** for caching and session management. To handle the high-volume telemetry data generated by the IoT device network, the system employs an event-driven communication model via **Apache Kafka**.

### System Stack
- **Backend:** Python 3.11 / Django 4.2 (REST Framework)
- **Database:** PostgreSQL 15 (RDS)
- **Caching/Messaging:** Redis 7.0 / Apache Kafka 3.5
- **Deployment:** AWS ECS (Elastic Container Service) using Fargate
- **Infrastructure:** AWS VPC, Route 53, Application Load Balancer (ALB)
- **Security:** ISO 27001 Compliant Environment

### Architecture Diagram (ASCII Representation)
```text
[IoT Devices] ----(MQTT/TLS)----> [IoT Gateway] 
                                       |
                                       v
[AWS ALB] <-------------------- [Kafka Cluster]
    |                                  |
    |          +-----------------------+-----------------------+
    |          |                       |                       |
    v          v                       v                       v
[Auth Service] <---> [Device Mgr Service] <---> [Reporting Service] <---> [Billing Service]
    |                        |                       |                       |
    +-----------+------------+-----------+-----------+-----------+-----------+
                |                        |                       |
                v                        v                       v
        [PostgreSQL DB]             [Redis Cache]          [S3 Bucket (PDFs)]
```

### Data Flow Description
1. **Ingestion:** IoT devices push telemetry and state changes to the IoT Gateway.
2. **Streaming:** The Gateway publishes events to Kafka topics (e.g., `device.heartbeat`, `device.alert`).
3. **Processing:** Microservices subscribe to these topics. The Device Manager updates the current state in PostgreSQL, while the Reporting Service aggregates data for the scheduled CSV/PDF exports.
4. **Consumption:** The customer-facing API queries the PostgreSQL and Redis layers to serve data to the frontend dashboard.

## 3. DETAILED FEATURE SPECIFICATIONS

### Feature 1: Customer-Facing API (Critical - Launch Blocker)
**Status:** In Review | **Priority:** Critical
The API serves as the primary interface for third-party integrations and the frontend dashboard. It must support strict versioning to ensure that legal clients using older integrations do not face breaking changes.

**Specifications:**
- **Versioning:** The API will utilize URI versioning (e.g., `/api/v1/...`). When a new version is released, the previous version must be supported for a minimum of 6 months.
- **Sandbox Environment:** A dedicated sandbox environment (`sandbox-api.wayfinder.oakmount.com`) will be provided. This environment uses a mirrored schema of the production database but contains anonymized data. It allows developers to test Webhooks and API calls without affecting real-world device telemetry.
- **Authentication:** All requests must be authenticated via OAuth2.0 Bearer tokens.
- **Rate Limiting:** Implemented via Redis. Standard tier: 1,000 requests/hour; Premium tier: 10,000 requests/hour.
- **Documentation:** Auto-generated via Swagger/OpenAPI 3.0.

### Feature 2: Customizable Dashboard (Low - Nice to Have)
**Status:** In Progress | **Priority:** Low
A web-based UI that allows users to visualize their IoT network health and legal asset tracking.

**Specifications:**
- **Widget Engine:** A grid-based layout system using React-Grid-Layout. Users can add, remove, and resize widgets.
- **Available Widgets:** 
    - *Device Health Gauge:* Real-time CPU/Memory usage of IoT nodes.
    - *Alert Feed:* A scrolling list of critical errors sorted by severity.
    - *Geospatial Map:* A Leaflet.js integration showing the physical location of legal hardware.
    - *Billing Summary:* A snapshot of current month's accumulated costs.
- **Persistence:** Dashboard configurations (widget positions, sizes, and visibility) are stored as JSON blobs in the `user_preferences` table in PostgreSQL.
- **Interactivity:** Widgets must be "drill-down" capable; clicking a widget opens the detailed view for that specific device or report.

### Feature 3: PDF/CSV Report Generation (Medium)
**Status:** Complete | **Priority:** Medium
The system must generate audit-ready reports for legal compliance, detailing device uptime and event logs.

**Specifications:**
- **Generation Engine:** Uses `ReportLab` for PDF generation and `Pandas` for CSV exports.
- **Scheduling:** A Celery beat worker manages the schedule. Users can set reports to be generated Daily, Weekly, or Monthly.
- **Delivery:** Reports are uploaded to an S3 bucket with a time-limited pre-signed URL. This URL is then emailed to the user via Amazon SES.
- **Templates:** Support for three legal-standard templates: "Compliance Audit," "Operational Health," and "Billing Reconciliation."
- **Filtering:** Reports must allow for date-range filtering and device-group filtering.

### Feature 4: SSO Integration (Critical - Launch Blocker)
**Status:** In Design | **Priority:** Critical
To meet ISO 27001 standards and Oakmount Group's corporate security policy, the system must integrate with existing identity providers.

**Specifications:**
- **SAML 2.0:** Integration with Azure AD for internal Oakmount employees.
- **OIDC (OpenID Connect):** Support for Google Workspace and Okta for external legal partners.
- **User Provisioning:** Just-In-Time (JIT) provisioning; users are created in the Wayfinder database upon their first successful SSO login based on the claims passed in the token.
- **Role Mapping:** SAML assertions must map to internal roles: `Admin`, `Editor`, `Viewer`.
- **Session Management:** Redis-backed session store with a maximum TTL of 8 hours, requiring re-authentication.

### Feature 5: Offline-First Mode (Low - Nice to Have)
**Status:** In Design | **Priority:** Low
Designed for legal technicians working in "dead zones" (e.g., courthouse basements) where connectivity is intermittent.

**Specifications:**
- **Client-Side Storage:** Uses IndexedDB via a Service Worker to cache the current state of the device network.
- **Background Sync:** Implements the Background Sync API. When the device regains connectivity, the service worker pushes a queue of "pending actions" to the `/api/v1/sync` endpoint.
- **Conflict Resolution:** A "Last-Write-Wins" strategy is applied by default, utilizing a `updated_at` timestamp on all records. In cases of critical data conflict, the system flags the record for manual review.
- **Data Payload:** Only essential telemetry and state changes are synced to minimize bandwidth during reconnection.

## 4. API ENDPOINT DOCUMENTATION

All endpoints require an `Authorization: Bearer <token>` header. Base URL: `api.wayfinder.oakmount.com`.

### 1. Get Device Status
- **Endpoint:** `GET /api/v1/devices/{device_id}`
- **Description:** Retrieves the current status and telemetry of a specific IoT device.
- **Response (200 OK):**
  ```json
  {
    "device_id": "WF-9901",
    "status": "online",
    "last_seen": "2023-10-24T14:00:00Z",
    "battery_level": 85,
    "firmware_version": "2.1.0"
  }
  ```

### 2. Update Device Configuration
- **Endpoint:** `PATCH /api/v1/devices/{device_id}/config`
- **Description:** Updates the configuration parameters (e.g., heartbeat interval) of a device.
- **Request Body:**
  ```json
  {
    "heartbeat_interval": 300,
    "logging_level": "DEBUG"
  }
  ```
- **Response (200 OK):** `{"status": "updated", "timestamp": "2023-10-24T14:05:00Z"}`

### 3. Create Report Schedule
- **Endpoint:** `POST /api/v1/reports/schedule`
- **Description:** Sets up a recurring PDF/CSV report delivery.
- **Request Body:**
  ```json
  {
    "report_type": "compliance_audit",
    "frequency": "weekly",
    "email": "compliance@oakmount.com",
    "format": "pdf"
  }
  ```
- **Response (201 Created):** `{"schedule_id": "SCH-123", "next_run": "2023-10-31T00:00:00Z"}`

### 4. List All Alerts
- **Endpoint:** `GET /api/v1/alerts`
- **Description:** Returns a list of all active alerts across the network.
- **Response (200 OK):**
  ```json
  [
    {"alert_id": "AL-501", "severity": "critical", "message": "Device WF-102 disconnected", "timestamp": "2023-10-24T12:00:00Z"},
    {"alert_id": "AL-502", "severity": "warning", "message": "Low battery on WF-202", "timestamp": "2023-10-24T12:10:00Z"}
  ]
  ```

### 5. SSO Initiation
- **Endpoint:** `GET /api/v1/auth/sso/init`
- **Description:** Redirects the user to the selected SAML/OIDC provider.
- **Request Parameter:** `?provider=azure_ad`
- **Response (302 Redirect):** Redirects to the Identity Provider (IdP) login page.

### 6. Sync Offline Data
- **Endpoint:** `POST /api/v1/sync`
- **Description:** Accepts a batch of offline events for processing.
- **Request Body:**
  ```json
  {
    "sync_id": "sync-abc-123",
    "events": [
      {"device_id": "WF-9901", "event": "state_change", "value": "offline", "timestamp": "2023-10-24T10:00:00Z"},
      {"device_id": "WF-9901", "event": "state_change", "value": "online", "timestamp": "2023-10-24T10:15:00Z"}
    ]
  }
  ```
- **Response (200 OK):** `{"processed_events": 2, "conflicts": 0}`

### 7. Get Billing Summary
- **Endpoint:** `GET /api/v1/billing/summary`
- **Description:** Retrieves the current expenditure for the IoT network.
- **Response (200 OK):**
  ```json
  {
    "current_month_cost": 1250.00,
    "currency": "USD",
    "billing_cycle_end": "2023-10-31"
  }
  ```

### 8. Delete Device
- **Endpoint:** `DELETE /api/v1/devices/{device_id}`
- **Description:** Decommissions a device and archives its telemetry data.
- **Response (204 No Content):** Empty body.

- `pref_id` (UUID, PK)

## 6. DEPLOYMENT AND INFRASTRUCTURE

### Environment Strategy
Wayfinder utilizes three distinct environments to ensure stability and security.

#### 1. Development (Dev)
- **Purpose:** Daily iteration and feature testing.
- **Infrastructure:** Smallest possible ECS cluster, shared PostgreSQL instance.
- **Deployment:** Triggered on merge to `develop` branch via a basic CI script.
- **Data:** Mocked data and sanitized subsets of production.

#### 2. Staging (Staging)
- **Purpose:** Pre-production validation, QA testing, and Stakeholder UAT.
- **Infrastructure:** Mirrored configuration of Production.
- **Deployment:** Manual trigger by the DevOps lead upon approval of the release candidate.
- **Data:** Anonymized production clones. This environment is used specifically to test the SSO integrations and Third-Party API rate limits.

#### 3. Production (Prod)
- **Purpose:** Live environment for Oakmount Group operations.
- **Infrastructure:** Multi-AZ ECS Fargate deployment for high availability. ISO 27001 certified AWS region.
- **Deployment:** Manual deployment performed by the single DevOps person.
- **Security:** Strict VPC isolation. Only the ALB is exposed to the public internet via Port 443.

### The "Bus Factor" Problem
Currently, the project has a **Bus Factor of 1**. All deployment scripts, CI/CD pipelines, and infrastructure-as-code (Terraform) are managed by a single DevOps resource. This creates a critical risk. To mitigate this, the project lead is pursuing the hiring of a contractor to document the deployment process and implement a more automated GitHub Actions workflow to reduce reliance on a single individual.

## 7. TESTING STRATEGY

### Unit Testing
- **Tooling:** `pytest`
- **Coverage Target:** 80% of business logic.
- **Focus:** Testing individual Django views, Kafka producer logic, and report generation helpers.
- **Execution:** Run on every commit via GitHub Actions.

### Integration Testing
- **Tooling:** `Postman / Newman`
- **Focus:** Validating the communication between microservices via Kafka and the interaction between the API and PostgreSQL.
- **Specific Test Case:** Ensuring that an alert triggered in the `Device Manager Service` correctly populates the `alerts` table and triggers a notification in the `Reporting Service`.

### End-to-End (E2E) Testing
- **Tooling:** `Cypress`
- **Focus:** High-priority user journeys:
    1. User logs in via SSO $\rightarrow$ Navigates to Dashboard $\rightarrow$ Views Device Health.
    2. User creates a report schedule $\rightarrow$ System generates PDF $\rightarrow$ User receives email.
    3. Device goes offline $\rightarrow$ Alert appears on dashboard in real-time.

### QA Role
A dedicated QA engineer is assigned to the project. The QA engineer is responsible for maintaining the test suite and performing "Regression Testing" every time a new feature is merged into the `staging` environment.

## 12. APPENDICES

### Appendix A: The "God Class" Refactoring Plan
The current `SystemManager` class is 3,000 lines long and handles:
1. User Authentication
2. System Logging
3. Email Dispatch
4. Device State Management

**Refactoring Strategy:**
- **Step 1:** Extract `AuthManager` $\rightarrow$ Move to a dedicated `auth_service.py`.
- **Step 2:** Extract `Logger` $\rightarrow$ Use Python's standard `logging` module integrated with AWS CloudWatch.
- **Step 3:** Extract `EmailDispatcher` $\rightarrow$ Create a `notification_service.py` utilizing Celery tasks.
- **Step 4:** Reduce `SystemManager` to a simple coordinator that delegates tasks to these services.

### Appendix B: ISO 27001 Compliance Checklist for Wayfinder
To maintain certification, the following must be verified monthly:
- [ ] All data at rest in PostgreSQL and S3 is encrypted using AES-256.
- [ ] All data in transit is encrypted via TLS 1.3.
- [ ] Access logs for the production environment are retained for 365 days.
- [ ] Quarterly password rotations are enforced for all non-SSO administrative accounts.
- [ ] A documented disaster recovery plan is tested every 6 months (snapshot restoration).
